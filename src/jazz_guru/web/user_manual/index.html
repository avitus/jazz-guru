jazz-guru · user manual

A walk through the architecture of a local Claude agent harness for music + text work — MusicXML, MIDI, WAV. Every concept maps to a file you can open; every diagram maps to code you can run.

Companion to docs/architecture.pdf and CLAUDE.md. Rendered live from the running server at /user_manual/.

00Introduction

jazz-guru is a local agent. You run it on your laptop, point it at a Claude API key, and it composes, arranges, analyzes, and discusses music with you across long-running sessions that remember what worked.

What it is, in one paragraph

The agent is a Python package distributed as three console scripts — jazz-guru (CLI), jazz-guru-server (FastAPI + WebSocket), and jazz-guru-worker (RQ worker for offline reflexion). It uses Anthropic Claude as its reasoner, Postgres + pgvector for durable state and semantic memory, and a handful of host binaries (FluidSynth, sfizz, liquidsfz, ffmpeg) for audio rendering. There is no Docker; everything runs out of a .venv on the host.

Why it exists

Off-the-shelf chat clients forget. They cannot author their own tools, cannot rehearse against a regression suite, cannot pour what they learned in last week's session into next week's. jazz-guru does all of that. The goal is not "another LLM wrapper" — it is a small, observable harness where every perception, plan, action, and observation goes into a JSONL trace and a Postgres event log you can replay, judge, and learn from.

Quick start

make setup        # brew deps + venv + DB + alembic
make migrate      # alembic upgrade head
make server       # FastAPI + websocket on 127.0.0.1:8000
make worker       # optional: RQ worker (needs Redis)

.venv/bin/jazz-guru info             # show resolved config
.venv/bin/jazz-guru new-session       # → UUID
.venv/bin/jazz-guru chat "hello"      # one-shot chat
.venv/bin/jazz-guru tui               # Textual UI (server must be running)
i

What this manual covers. Sixteen sections that mirror the architecture PDF page-for-page, plus a short reference appendix. Each section opens with a one-line summary, links to the actual source files, and explains the design decisions behind what you'll see in the code.

How to read this manual

  • Skim first. The system overview (§1) gives you the thirty-second tour. Stop there if you only want a mental model.
  • Trace one turn. The turn lifecycle (§2) walks a single chat message all the way through — the most useful section for figuring out where to set a breakpoint.
  • Then drill. The remaining sections cover individual subsystems and stand alone; jump to whatever you're hacking on.

01System overview

Clients invoke the FastAPI server; the server drives the agent loop; each turn touches Postgres, the workspace filesystem, and external services.

Bird's-eye view. Clients hit the server; the server drives the agent core; the core leans on tools, persistence, and external services.

Five moving parts

clients
Anything that speaks to 127.0.0.1:8000 — the CLI, the Textual TUI, the static web UI at /ui/, a typed Python SDK, the trace viewer, or your own HTTP client. The server is what gives them all a uniform surface.
FastAPI server
One file: src/jazz_guru/server.py. Hosts the REST routes, the WebSocket for streaming turn events, the static UI, and an optional X-API-Key middleware. The lifespan handler also starts/stops the MCP manager.
agent core
The textbook perceive → plan → act → observe loop, in src/jazz_guru/harness/loop.py. One call to AgentLoop.step() is one user turn. Sections 2–3 walk it in detail.
tools & sandbox
Every action the agent can take is a registered tool in src/jazz_guru/actions/tools/. The agent can also author its own tools via the meta-tools — see §5.
persistence
Postgres + pgvector for sessions, turns, events, memory, playbook, generated tools, eval runs. The workspace/ directory holds session artifacts, dynamic tool source, state snapshots and JSONL traces. Redis + RQ are optional — only needed for async reflexion.

External dependencies

  • Anthropic Claude — reached through src/jazz_guru/llm.py, which wraps the SDK with tenacity retries on ConnectionError, Timeout, 429 and 5xx, and accumulates token / cost usage into LLMUsage.
  • Voyage AI embeddings — only when VOYAGE_API_KEY is set. With it blank, the memory store falls back to a deterministic sha256-derived float32 vector of the configured dimension. This is what the test suite uses; no network in CI.
  • Web Search API — backs the web_search tool.
  • Audio engines — FluidSynth, sfizz, liquidsfz, and ffmpeg, all brew-installed. render_midi dispatches to whichever one a preset names.

02Turn lifecycle

What actually happens when the user sends one chat message. Useful as a breakpoint guide: every step here corresponds to a callable you can step into.

One AgentLoop.step(message) call, end to end

  1. Record the user turn. A Turn row with role='user' is inserted into Postgres. The async DB session is acquired through db.session_scope().
  2. Perceive. Three reads in parallel-ish:
    • memory.search(query, k=retrieve_k, session_id=...) against memory_items with a pgvector cosine query.
    • Top-N PlaybookEntry rows by score — global strategies refined by reflexion (§8).
    • state.load_latest(sid) — reads workspace/state/<sid>/latest.json and walks workspace/sessions/<sid>/ to enumerate artifacts.
  3. Plan. ContextBuilder.build(...) assembles the system prompt by concatenating: goal.md + goal.yaml + _TOOL_CREATION_HINT + StateDoc.render_markdown() + playbook entries + retrieved memory + (rolling-history summary).
  4. Hydrate dynamic tools. tool_store.load_all_specs() pulls all globally-published rows from generated_tools into this session's DynamicRegistry, which is merged with the static registry. ActionController.allowed is recomputed against the merged view through the policy filter.
  5. Act / observe. ActionController.run(messages, tools, on_event) drives the Anthropic tool_use loop. Each round:
    1. llm.complete(system, messages, tools, tool_choice='auto') and emit llm_request / llm_response.
    2. If stop_reason != 'tool_use', return.
    3. For each tool_use block: emit tool_use, invoke the tool through the merged registry, append the result to messages, emit tool_result.
    4. Check the budget. policy.budgets.per_turn.tool_calls doubles as max_rounds.
  6. Persist + reply. Insert an assistant Turn row plus Event rows. Write a summary MemoryItem for the exchange. Write workspace/state/<sid>/latest.json. Emit turn_end. Return the response to the caller.
Each step emits an event; subscribers fan it out to JSONL traces, the WebSocket, and the Postgres event log.
i

Where to set a breakpoint. If you're chasing a bug in tool dispatch, start at ActionController.run in actions/controller.py. If you're chasing a bug in what the model sees, start at ContextBuilder.build in context/builder.py. If you're chasing a memory/playbook regression, start at AgentLoop._retrieve_memory in harness/loop.py.

03Agent core internals

Four collaborators inside the loop: the loop itself, the context builder, the action controller, and the LLM wrapper. Every chat-style call passes through this seam.

The loop — harness/loop.py

AgentLoop holds the long-lived per-session state: the SessionHandle, the TraceWriter, the DynamicRegistry, a BackgroundJobRegistry (renders started in turn 1 must remain inspectable in turn 5), the ActionController, and the MemoryStore. Constructor arguments let tests inject mocks. step() is the only method callers should reach for.

Context builder — context/builder.py

Given a BuildInputs bundle (goal, state doc, playbook entries, memory hits, history), the builder produces the final (system, messages) pair handed to llm.complete. The system prompt is laid out as:

GoalConfig.render_system_block() + _TOOL_CREATION_HINT + StateDoc.render_markdown() + playbook (top-N) + retrieved memory + history summary

The tool-creation hint is the only static block we bake into the prompt; it tells the model how to use tool_create / tool_publish / tool_promote_to_source. Everything else is configurable through config/goal.{md,yaml}.

Action controller — actions/controller.py

The controller owns three responsibilities: filtering the merged tool registry against policy, running the tool_use loop against the Anthropic API, and emitting events. The filtering looks like: 

for spec in registry.specs():
    p = policy.for_tool(spec.name)
    if p.mode != 'allow':
        drop
    elif p.feature_flag and not os.getenv(p.feature_flag):
        drop
    else:
        keep

The result is the tools=[...] JSON-schema list handed to Claude. The loop runs at most policy.budgets.per_turn.tool_calls rounds, then bails with an error event if the model still wants to keep calling tools.

LLM wrapper — llm.py

A thin wrapper over the Anthropic SDK. It exists for three reasons:

  • Retries. Tenacity wraps the call with exponential backoff on ConnectionError, Timeout, 429, and 5xx.
  • Usage accounting. Every call accumulates an LLMUsage record (input tokens, output tokens, USD cost). The loop sums these into the Turn row.
  • Feature pass-through. Extended thinking, streaming, tool_choice, and similar SDK options pass through verbatim.
✓

If you want to change which model is used, set ANTHROPIC_MODEL in .env. The harness defaults to a modern Claude; older retired models are not auto-replaced.

04Tool system & catalogue

A process-global ToolRegistry is merged each turn with a per-session DynamicRegistry, then filtered by policy and dispatched by the controller.

The static registry

Tools are Python functions registered through a decorator: 

@registry.register(
    name="fs_read",
    description="Read a UTF-8 text file from the session workspace.",
    input_model=FsReadInput,
)
async def fs_read(**kwargs) -> dict:
    ...

register_all() in actions/registry.py imports every module under actions/tools/; the decorators fire and the registry is populated. It's idempotent — safe to call from tests.

Built-in catalogue

The current install ships with the tools below. They are organised loosely into families; the policy file lets you flip whole families on or off via toolsets.

FamilyToolsWhat they do
filesystem fs_read, fs_write, fs_list Workspace-scoped reads, writes, and listings. Refuses paths outside the session.
shell / python shell, python_exec Subprocess escape hatches. Sandbox-wrapped when JG_OS_SANDBOX=1.
code editing code_gen, code_edit, patch Generate, find-and-replace, or multi-strategy patch files.
network http_get, http_post, web_search SSRF-guarded HTTP plus web search.
music · symbolic music_xml_info, music_xml_from_tinynotation, music_xml_transpose, midi_info, midi_from_notes Inspect, build, transpose MusicXML; inspect and build MIDI.
music · render & presets render_midi, preset_list, preset_get, preset_upsert, preset_delete MIDI→WAV through FluidSynth / sfizz / liquidsfz, ffmpeg post chain, agent-mutable preset surface.
perception audio_analyze, vision, tts librosa-based audio analysis, image understanding via vision, optional TTS (FEATURE_TTS=1).

Authoritative list: .venv/bin/jazz-guru tools. Source files: src/jazz_guru/actions/tools/.

Policy filter — config/policy.yaml

The policy file is the agent's permission system. Three things live there:

per-tool entries
Each tool can set mode: allow|deny, a scope hint, a timeout_sec, a memory cap, a max response size, and a feature_flag env-var gate.
toolsets
Named groups (core, fs, shell, web, code, music, audio, vision, meta). Per-name entries always win.
budgets
per_turn.tool_calls — doubles as the controller's max_rounds. per_turn.wall_clock_sec, per_session.tokens, per_session.usd.

Sandbox helpers — actions/sandbox/

  • resolve_in_workspace(path, sid) — strict: only paths inside workspace/sessions/<sid>/. Use this for any write and any user-supplied path that should stay session-scoped.
  • resolve_in_safe(path, sid) — read-oriented: same plus data/ plus JG_SAFE_EXTRA_PATHS. Use this for tools that legitimately read project data (e.g. preset lookups).

ToolContext — actions/context.py

Tools never receive the session id as a function argument. Instead, the controller stashes session_id and turn_idx in a contextvar before invoking each tool; current() reads them back. Tools that need to write into the session workspace pull the id from this contextvar — that's how resolve_in_workspace finds the right directory.

!

Never add a path argument that bypasses the resolvers. shell and python_exec are subprocess escape hatches by design — there is no path filtering inside the child process. Set JG_OS_SANDBOX=1 to additionally wrap them and dynamic-tool subprocesses with sandbox-exec -f data/sandbox/jg.sb; see §13.

05Dynamic tools (3 tiers)

The agent can author its own tools mid-session. Three promotion tiers trade off reversibility against permanence: session-local → globally published → first-class source.

Tier 1 · session-only

tool_create(name, description, input_schema, source)

Validates, writes workspace/sessions/<sid>/tools/<name>.py, attaches a DynamicSpec to the session's DynamicRegistry, recomputes controller.allowed.

Vanishes when the session ends — unless promoted.

Tier 2 · globally published

tool_publish(name)

UPSERTs the spec into the generated_tools table with scope='global'. Every future session sees it automatically when _hydrate_dynamic_registry runs — no restart needed.

Reversible at runtime via tool_remove (soft-delete, deprecated=True).

Tier 3 · first-class source

tool_promote_to_source(name)

Writes the file into src/jazz_guru/actions/tools/<name>.py. The directory glob picks it up on next import.

Needs a server / worker restart to take effect. Tier 3 changes the codebase; tier 2 stays runtime-reversible.

Execution model

The default execution mode for a dynamic tool is a fresh python -I <tool.py> subprocess:

cwd
workspace/sessions/<sid>/
stdin
JSON kwargs
stdout
JSON result
timeout
from the python_exec policy entry
sandbox
wrapped with sandbox-exec when JG_OS_SANDBOX=1
inproc fallback
for trusted helpers that need to share process state

Hydration on every turn

At the top of AgentLoop.step(), _hydrate_dynamic_registry() calls store.load_all_specs(), pulling every global row from generated_tools into the per-session DynamicRegistry. That registry is merged with the static one before the policy filter runs.

Invocation path

LLM emits tool_use → ActionController dispatch → merged registry lookup → Pydantic validate(input) → subprocess (python -I) / inproc call → JSON tool_result → on_event(tool_result)
i

Why tier 2 vs tier 3 is deliberate. Tier 2 keeps experiments contained — you can tool_remove them at runtime, and bad ones are isolated by the subprocess model. Tier 3 is for tools the operator has decided deserve to be bound into the source. Both are surfaced to Claude through the same tool_use interface; the operator decides when to bake one in.

06Database schema

Postgres + pgvector. Eight tables managed by Alembic. All async access through db.session_scope(); the synchronous URL exists only for migrations.

Tables

TableHoldsNotable columns
sessions One row per conversation. id (UUID), goal_profile, title, meta JSON, created_at.
turns Every user / assistant / tool / system message. session_id FK, idx (per-session ordinal), role, content JSON, tokens_in/out, cost_usd, started_at, ended_at.
events The event stream (one row per controller emission). session_id, optional turn_id, type (indexed; §12), payload JSON, ts (indexed).
snapshots References to workspace/state/*/turn_*.json on disk. session_id, optional turn_id, path, sha256.
memory_items Long-term semantic memory. kind (lesson, fact, ...), text, meta, embedding pgvector sized to settings.embedding_dim, score.
playbook_entries Global strategies (no session FK), refined by reflexion. scope (e.g. voicing, rhythm, workflow), text, score, meta.
generated_tools Globally-published dynamic tools. name UNIQUE, input_schema JSON, source, sha256, scope, owner_session_id, version, deprecated.
eval_runs One row per task per regression run. task_id, session_id, score, rubric JSON, notes.

ORM models: src/jazz_guru/state/schema.py. Migrations: alembic/.

Access & operations

async with db.session_scope() as s:
    s.add(turn)
    await s.commit()
  • pgvector required. CREATE EXTENSION IF NOT EXISTS vector; — the embedding column is sized off settings.embedding_dim, so Voyage and hash-stub must agree on dimensionality.
  • Two URLs. DATABASE_URL (asyncpg) for the app, DATABASE_URL_SYNC for alembic only.
  • Migrations. make migrate runs alembic upgrade head.

07Memory subsystem

How the agent recalls. A pgvector cosine search backed by Voyage embeddings (or a deterministic hash-stub), a rolling-history summarizer, and a StateDoc projection that lands in the system prompt.

MemoryStore — memory/store.py

Two async methods: search(query, k, session_id, kind) and write(text, kind, score, meta). Search runs: 

emb = await embeddings.embed(query)
SELECT *, 1 - (embedding <=> :emb) AS sim
FROM memory_items
[WHERE kind = :kind AND ...]
ORDER BY sim DESC
LIMIT :k

Failures — e.g. pgvector unavailable, embeddings client throws — are caught and logged. An embedding outage does not break a turn; the agent just gets fewer hits.

Embeddings — memory/embeddings.py

The Voyage client is wrapped behind a single async embed(text) surface. When VOYAGE_API_KEY is blank, a deterministic sha256-derived float32 vector of settings.embedding_dim is used instead. The fallback is what the test suite relies on, so the suite never touches the network.

StateDoc — state/externalize.py

The "self-model" that gets injected into the system prompt: 

@dataclass
class StateDoc:
    summary: str
    open_threads: list[str]
    artifacts: list[str]
    last_critique: str | None

load_latest(sid) reads workspace/state/<sid>/latest.json and enumerates artifacts by walking workspace/sessions/<sid>/. render_markdown() produces the block the ContextBuilder stitches into the prompt.

History compression

The summarizer in memory/summarizer.py takes the chat history above a configurable cutoff and replaces it with a short LLM-written summary held as a system note. This keeps the token budget bounded for long-running sessions without losing the gist.

Snapshots

  • Written every turn end and at reflexion completion.
  • Files: workspace/state/<sid>/latest.json (current) and turn_NNNNN.json (history).
  • Payload: summary, open_threads, artifacts, last_critique, last_plan, last_score.
  • A Snapshot row in Postgres references the path and sha256.

The read-path, in one diagram

ContextBuilder.build(...)
  ⋮ MemoryStore.search(query)  →  semantic hits (pgvector cosine)
  ⋮ SELECT PlaybookEntry ORDER BY score LIMIT N
  ⋮ state.load_latest(sid)  →  StateDoc.render_markdown()
  ⋮ summarizer.compact(history)  →  bounded history block
system_block + messages[] → llm.complete

08Distillation / Reflexion

Offline learning loop. Runs after a session, summarises the transcript, critiques it, and distils memory + playbook + snapshot updates that the next session will see automatically.

Two triggers

async (default)

POST /sessions/{id}/distill or .venv/bin/jazz-guru distill <sid>

Enqueues reflexion_job on the Redis / RQ queue. The make worker process picks it up. Caller returns immediately.

sync (inline)

POST /sessions/{id}/distill?sync=true or jazz-guru distill <sid> --sync

run_reflexion is invoked inline in the request handler — no Redis hop. Useful in tests and one-off runs.

The flow — distillation/reflexion.py

  1. summarize_transcript(session_id) — collapses Turn rows and key Event rows into a compact transcript.
  2. Build the reflexion prompt. Input: session goal block, transcript summary, current StateDoc, artifact list. Instruction: emit STRICT JSON only.
  3. llm.complete(...) with the strict JSON contract.
  4. Apply writeback (see below).
  5. Emit the reflexion event — subscribers like the trace writer and Postgres event log pick it up.

The JSON contract

{
  "score": 0.0,        // 0.0 - 1.0, clamped
  "critique": "...",
  "revised_plan": "...",
  "open_threads": ["...", "..."],
  "memory_writes": [
    { "text": "...", "kind": "lesson" }
  ],
  "playbook_entries": [
    { "scope": "voicing", "text": "...", "score": 0.7 }
  ]
}

Writeback targets

memory_writes
Insert MemoryItem rows (default score=0.7, kind from each entry).
playbook_entries
Upsert PlaybookEntry rows by (scope, text).
snapshot
Write a fresh workspace/state/<sid>/latest.json carrying the summary, open_threads, artifacts, last_critique, last_plan, last_score.

How reflexion lands in the next turn

The next AgentLoop.step() sees the writeback through the same code paths the agent uses every turn:

  • memory.search(query) picks up the new MemoryItem by semantic similarity.
  • The top-N PlaybookEntry read picks up the upserted strategy entry.
  • state.load_latest() carries summary / open_threads / last_critique into the system prompt via StateDoc.

09Eval / regression suite

Task replay plus LLM-as-judge. Each run produces a scored row in eval_runs you can trend across commits or branches.

Task format

Each YAML file under eval/tasks/ declares one task:

id
String, used as EvalRun.task_id.
prompt
The user message.
expected
Target behaviour / criteria for the judge.
setup
Optional fixtures (files copied into the session workspace).
rubric
List of dimensions (e.g. correctness, tool usage, brevity).

Runner — eval/runner.py

For each task:

  1. Create a fresh Session.
  2. Apply fixtures.
  3. Issue task.prompt via AgentLoop.step.
  4. Capture the assistant message plus the tool trace.
  5. Call judge(task, transcript).
  6. Insert an EvalRun row with (task_id, session_id, score, rubric, notes).

LLM-as-judge — eval/judge.py

The judge prompt bundles task.expected + task.rubric + transcript, then calls llm.complete with a strict JSON output: 

{
  "score": 0.0,
  "rubric": { "<dimension>": 0.0, ... },
  "notes": "..."
}

These land verbatim in EvalRun.score, .rubric, and .notes.

Trigger paths

  • .venv/bin/jazz-guru evalrun [--only <task-id>]
  • POST /eval/run (sync handler; you can route this through an RQ enqueue if you'd rather not block the request).
  • Any CI step that posts and polls for EvalRun rows.

10Audio rendering pipeline

render_midi picks an engine from the preset, renders to a temporary WAV, runs an ffmpeg post-process chain, and writes the final WAV into the session workspace.

Three engines

fluidsynth

SoundFont (.sf2 / .sf3); General MIDI program changes. Best for orchestral or GM banks.

sfizz

SFZ via sfizz_render. Modern sampler scripting; good when you want articulation control.

liquidsfz

SFZ via liquidsfz --no-jack --render. Alternative SFZ renderer; useful when sfizz misbehaves on a particular bank.

All three are brew-installed; paths resolved through PATH. Source: actions/tools/render.py.

The preset surface — data/instruments.yaml

Presets are the agent-mutable surface that names a render configuration. The only sanctioned read/write path is src/jazz_guru/presets.py (mtime-cached; render_midi reloads on every call). Library paths resolve against JG_INSTRUMENTS_ROOT (default ~/.local/share/jazz-guru/instruments). 

version: 1
default: <preset name>
presets:
  <name>:
    engine: fluidsynth | sfizz | liquidsfz
    library: path/to/sf2|sfz
    description: ...
    post:
      lowpass_hz:    float?
      lowpass_q:     float?
      vibrato_hz:    float?
      vibrato_depth: float?
      gain_db:       float?
      normalize:     bool

Preset tools

The agent mutates presets through dedicated typed tools, never via fs_* or shell:

  • preset_list() — names + descriptions + the current default.
  • preset_get(name) — full preset record.
  • preset_upsert(name, spec) — pydantic-validated, atomic-write.
  • preset_delete(name) — refuses if the preset is currently the file default.

ffmpeg post chain

Applied after the engine render. Each field overrides only the corresponding stage; everything else falls through to the preset defaults on a per-render basis.

Stageffmpeg expressionWhat it does
lowpass lowpass f=<lowpass_hz> [q=<lowpass_q>] Muffle / smooth tone.
vibrato vibrato f=<vibrato_hz> d=<vibrato_depth> LFO pitch wobble.
volume volume <gain_db>dB Loudness adjust.
loudnorm loudnorm I=-16 TP=-1.5 LRA=11 Broadcast-style normalization (only when normalize=true).

What you get back

render_midi writes the final WAV into workspace/sessions/<sid>/ and returns { path, duration_sec, sha256 }.

11Configuration sources

Three loaders, all funnelled through src/jazz_guru/config.py with an lru_cache. A restart is needed to pick up changes in long-running processes (server, worker).

.env — pydantic-settings

Loaded by pydantic_settings.BaseSettings. The relevant fields:

JG_API_KEY""Set → /chat etc. require X-API-Key.

config/goal.{md,yaml} — the agent's north star

goal.md is freeform prose (identity, working style, long-term arc); goal.yaml adds structured fields: 

profile: default
objectives:
  - id: make-music
    text: prefer producing playable artifacts over long descriptions
    weight: 1.0
constraints:
  - keep prose concise
success_criteria:
  - rendered .wav available within 2 turns
style:
  voice: ...
  jargon: ...
  notation_priority: ...

GoalConfig.render_system_block() concatenates both and prepends them to every system prompt.

config/policy.yaml — per-tool gates + budgets

Covered in §4. The fusion rule, summarised: 

ActionController.allowed = [
    spec
    for spec in registry.specs()
    if policy.for_tool(spec.name).mode == 'allow'
    and (not p.feature_flag or os.getenv(p.feature_flag))
]

Resolution order in Policy.for_tool: per-name entry → matching toolset → default.

How they fuse into the system prompt

goal.md + goal.yaml → static, in context/builder.py
StateDoc → per-session snapshot
playbook → top-N from playbook_entries
memory → MemoryStore.search()
history → summarizer.compact()
!

Cache caveat. get_settings is lru_cache'd. If you edit .env or any YAML in a long-running server or worker, restart the process — or call get_settings.cache_clear() from a debug shell.

12Observability

Every controller step emits an event. Three subscribers consume the stream: a JSONL trace writer, the WebSocket fanout, and the Postgres event log.

Event types — EventType in state/schema.py

The fanout

ActionController.on_event(event_type, payload, *, session_id, turn_id) calls every subscriber attached by AgentLoop / the server during step() construction. Three subscribers ship out of the box:

TraceWriter

logging/trace.py

Appends one JSON object per event to workspace/traces/<sid>.jsonl. Format: {"ts": iso, "type": ..., "turn_id": ..., "payload": ...}.

WebSocket

WS /ws/sessions/{id}/chat

Pushes every event, in order. Auth via subprotocol header: Sec-WebSocket-Protocol: bearer, <JG_API_KEY>. auth.require_ws() runs explicitly since Starlette middleware doesn't fire on WS.

Postgres event log

state/event_log.py

Inserts a row into events for every emission, indexed by type and ts. Queryable for replay, postmortems, telemetry.

Trace viewer — jazz-guru viewer

Tiny uvicorn app at :8765. Lists workspace/traces/*.jsonl, tails the chosen file in a browser, and pretty-prints each event by type. Useful when you're not running the full FastAPI server but still want a live view of an agent run.

structlog (app-level)

Configured at module import in logging/__init__.py. JSON or console renderer. Used for everything outside the event stream — setup errors, retries inside llm.complete, config loading, alembic migrations, FastAPI access logs via uvicorn.

Sandbox helpers

actions/sandbox/ exposes two resolvers:

  • resolve_in_workspace(path, sid) — strict, only paths inside workspace/sessions/<sid>/. Used for writes and any user-supplied path that should stay inside the session.
  • resolve_in_safe(path, sid) — read-oriented; same set plus data/ plus JG_SAFE_EXTRA_PATHS. Used by tools that legitimately read project data, like preset lookups.

Tools that touch project data outside the session expose a dedicated typed surface (see actions/tools/presets.py). Never require the agent to use fs_write / shell / python_exec against project data.

macOS sandbox profile — data/sandbox/jg.sb

When JG_OS_SANDBOX=1 the harness wraps shell, python_exec, and dynamic-tool subprocesses with: 

sandbox-exec -f data/sandbox/jg.sb \
  -D WORKSPACE=/abs/workspace/sessions/<sid> \
  -D DATA=/abs/data \
  <argv...>

Rules summary:

  • deny everything by default;
  • allow process startup (fork, exec, signal, mach-lookup, sysctl-read, ipc-posix-shm);
  • allow file-read against /usr/lib, /usr/share, /System/Library, Frameworks, /opt/homebrew, /usr/local, /private/var/db/dyld, pyenv + .venv install roots, plus {WORKSPACE} and {DATA};
  • allow file-write against {WORKSPACE}, /private/tmp, /private/var/folders;
  • allow network (so HTTP / web search tools still work; revoke this in your profile if you want full isolation).
i

The flag is opt-in. Off by default because the profile is macOS-specific (no Linux equivalent ships in this repo). When you turn it on, expect to add paths to the profile if your tools need libraries outside the allow-list.

Auth — auth.py

Off unless JG_API_KEY is set in the env. When on:

  • HTTP: every request must carry X-API-Key: <value> in the header. Missing or wrong → 401.
  • WebSocket: middleware does not run for WS. Each handler calls auth.require_ws(token), reading the token from the Sec-WebSocket-Protocol header. Convention: client sends ["bearer", "<token>"]; the server echoes "bearer" as the negotiated subprotocol.
  • Exempt paths: /, /health, /docs, /redoc, /openapi.json, /favicon.ico, /ui, /ui/*, /static/*.

15MCP & subagents

Two surfaces the architecture diagram only hints at: Model Context Protocol integration for plugging in external tool servers, and delegate_task for spawning child agents.

MCP — actions/mcp/

The Model Context Protocol lets the agent reach tools that live in a separate process — another binary, a remote service, an operator-built sidecar. The harness starts and stops MCP servers alongside the FastAPI lifecycle through MCPManager:

  • Configuration: config/mcp.yaml.
  • Lifespan: _lifespan in server.py calls mgr.start_all() on startup and mgr.stop_all() on shutdown; failures are logged but do not abort startup — a misconfigured MCP server should not prevent the rest of the agent from running.
  • Status & reload: GET /mcp/status and POST /mcp/reload.

Subagent delegation — delegate_task

Implemented in actions/tools/delegate.py. Spawns a child agent with its own conversation history for an isolated sub-task. The child reports back through the same tool-result interface; the parent continues with the result folded into its messages. Useful when one branch of work would otherwise pollute the parent context (large research sweeps, exhaustive enumeration over options).

Background jobs — actions/jobs.py

Long-running tool work (e.g. a multi-minute render) is dispatched into a BackgroundJobRegistry that persists for the lifetime of the AgentLoop — i.e. the session. A render started in turn 1 remains inspectable in turn 5. Tools can spawn a job, return immediately with a handle, and let the agent poll later through a follow-up tool call.
